Product Decision Record

Baddiecore: Linux-first headless CMS inspired by self-hosted Sitecore

This document captures the current product direction for Baddiecore before implementation starts. It favors a focused Sitecore-familiar MVP: content tree, visual page editing, YAML templates, draft/published states, workflow, and a REST layout API.

Status Initial MVP direction
Date 2026-06-20
Primary deployment target Linux containers and Linux servers

Product Position

Decision Baddiecore is a self-hosted, headless CMS built on modern .NET, intended to run cleanly on Linux.
Decision The product should feel familiar to Sitecore developers and editors, but it does not need direct Sitecore compatibility.
Decision The immediate goal is a working showcase and technical proof, not a commercial CMS product.
Core thesis: prove that a Sitecore-like self-hosted headless CMS can be built with modern .NET and deployed on Linux.

MVP Scope

Included

  • Corporate or marketing demo site.
  • Content tree explorer.
  • SolidJS/Vite CMS dashboard served by the ASP.NET Core host.
  • Visual page editor with iframe preview.
  • Field editing for pages and components.
  • Add, remove, and reorder page components without drag-and-drop in the first version.
  • Pages, text content, component content, and media such as images.
  • Draft and published states.
  • Publish history and basic version history.
  • Manual workflow approval before publishing.
  • Role-based permissions for admin, content-admin, and author users.
  • Controller-based ASP.NET Core API endpoints.
  • REST layout API for frontends.
  • YAML-defined templates owned by developers and synced into the CMS.

Deferred

  • Multilingual content.
  • Scheduled publishing.
  • Drag-and-drop page editing.
  • Search.
  • Personalization.
  • Analytics, campaigns, goals, A/B tests, and profile tracking.
  • Direct migration or API compatibility with Sitecore.

Application Shell

Baddiecore will use ASP.NET Core as the backend host and serve the CMS dashboard from the same application in production. The dashboard itself should be a SolidJS app built with Vite.

Decision Backend endpoints should use ASP.NET Core controllers rather than minimal API endpoints.
Decision The CMS dashboard should be written as a SolidJS/Vite single page app.
Decision In production, the Vite build output can be emitted or copied into wwwroot/cms and served by the ASP.NET Core app.
Decision During local development, the SolidJS dev server can run separately and call the ASP.NET Core API through a proxy or configured API base URL.

Editing Experience

The first editor experience should be Sitecore-like, centered on a content tree and an Experience Editor-style page preview. It should not attempt true inline editing in the MVP.

Decision Editors select a page from the content tree, preview the real frontend page inside an iframe, and select page components from the preview.
Decision Editing a selected component opens a side panel or menu containing the component fields, commonly text fields.
Decision Editors can add, remove, and reorder components on a page, but drag-and-drop is outside the first version.

Iframe Preview

The CMS will use an iframe-based preview so the frontend remains bring-your-own. The CMS stores content and page layout; the frontend owns rendering, styles, behavior, and responsive output.

Decision The CMS editor shell embeds the frontend preview URL inside an iframe.
Decision Preview mode uses a token or similar access mechanism so the frontend can request draft content.
Decision The iframe and CMS shell communicate via postMessage for component selection and editor actions.
Decision Frontend components should expose stable rendering IDs, such as data-baddie-rendering-id, so clicks can map back to CMS data.
CMS Editor Shell toolbar + workflow actions content tree iframe: https://frontend.local/preview?path=/about&token=... edit panel for selected rendering Frontend Preview fetches draft layout data from Baddiecore renders the real page sends selected rendering ID to the CMS shell with postMessage

Headless Layout API

The delivery contract should be REST-first and layout-oriented. The frontend asks for a page layout by path and mode, then renders the returned placeholders, renderings, fields, media, and metadata.

Decision Baddiecore should expose a REST layout API similar in spirit to Sitecore headless layout endpoints.
Decision The first API shape can be path-based, with preview/draft access controlled separately.
GET /api/layout?path=/about&mode=preview Response includes: page fields placeholders renderings/components component field values media URLs draft/published metadata version information

Templates And Renderings

Developers own template and rendering definitions. The CMS imports or syncs YAML definitions instead of treating the editor UI as the source of truth for template structure.

Decision Templates are represented as YAML files and stored in version control.
Decision The CMS can import or sync those YAML definitions.
Decision Editors create and modify content using templates, but do not change fundamental template structures in the MVP.
Decision A rendering is a named frontend component contract with editable fields, optional data source template, and allowed placeholders.
rendering: name: Hero componentKey: hero datasourceTemplate: HeroContent allowedPlaceholders: - main

Content Model

Decision Baddiecore keeps a Sitecore-familiar content tree model for the MVP.
Decision Content should support pages, component data, text fields, media items, and page composition data.
Decision The first demo should include common corporate site templates such as home page, standard page, article page, hero, rich text, CTA, image, and card grid.

Workflow And Publishing

The workflow is intentionally simple. Approval and publishing are separate actions so approved content is not automatically pushed live.

Draft Awaiting Approval Approved Published
Decision Authors can create and edit pages, then submit them for approval.
Decision Content-admins can approve, reject, publish, and unlock content.
Decision Admins can do everything, including developer-oriented and structural actions.
Decision Publishing should preserve version history. The MVP can start with draft and published states plus history on publish.

Roles

Admin

Full access. Most likely developers or system owners. Can manage everything, including structural configuration.

Content Admin

Can manage most content operations, including approval, publishing, and locks. Cannot change templates or fundamental user role structures.

Author

Can create and edit pages, manage assigned content, and submit work for approval.

Future Permissions

More granular item-level permissions can be added after the first content workflow is proven.

Open Questions

  • Which database should be the first supported persistence target: MySQL, SQL Server, PostgreSQL, or another option?
  • Should the first demo frontend be React, Next.js, ASP.NET Core, or another stack?
  • How should YAML template sync handle removed fields, renamed fields, and existing content values?
  • What is the smallest useful media library for the first demo?
  • Should the first version use a single app with authoring and delivery modes, or split CM/CD processes later?
Architecture decisions are intentionally deferred until the product shape is stable enough to justify them.
